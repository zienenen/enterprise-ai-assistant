# 一期研发任务分解清单

本文是 `enterprise-ai-assistant` 新仓库的一期研发派单依据。旧 `plan-management` 只作为业务逻辑、PPT 模板处理、字段习惯和历史问题参考，不再作为主线开发仓库。

## 0. 强制约束

- 顶层产品是“企业管理 AI 助手平台”，不是计划管理系统。
- 登录后第一屏必须是企业管理 AI 助手门户 `/portal`。
- 计划管理助手只是一期第一个完整落地的业务助手。
- 不允许把 `/api/plans/*` 作为新系统顶层 API。
- 计划管理助手 API 统一放在 `/api/assistants/plan/*`。
- 开发、测试、生产环境统一使用 PostgreSQL，不使用 SQLite。
- AI 工作流中心一期只做 demo 展示，不驱动真实业务执行。
- 每个任务完成后必须验证、提交并推送 GitHub。

## 2. 工程基线任务

### EAI-M0-01：后端 PostgreSQL 基线

目标：Django 后端、开发环境、测试环境统一使用 PostgreSQL。

实现范围：

- `backend/.env.example` 包含 PostgreSQL 配置。
- Django settings 只配置 PostgreSQL，不提供 SQLite 回退。
- 本地应用账号具备 `CREATEDB`，支持 Django test 创建测试库。
- 完成数据库迁移。

验收标准：

```powershell
cd backend
.\.venv\Scripts\python.exe manage.py check
.\.venv\Scripts\python.exe manage.py makemigrations --check --dry-run
.\.venv\Scripts\python.exe manage.py test
```

全部通过，且测试库由 PostgreSQL 创建。

### EAI-M0-02：Redis 与 Celery 基线

目标：建立异步任务基础能力。

实现范围：

- 配置 `REDIS_URL`。
- 建立 Celery app。
- 建立 smoke task。
- 建立任务状态接口或模型雏形。

验收标准：

- Celery worker 可启动。
- smoke task 可执行。

### EAI-M0-03：前端构建基线

目标：Next.js 前端可安装、可构建。

验收标准：

```powershell
cd frontend
npm.cmd install
npm.cmd run build
```

构建成功，不提交 `.next`、`node_modules`。

## 3. 平台级任务

### EAI-M1-01：认证与登录后跳转

实现范围：

- 后端提供登录、登出、当前用户接口。
- 前端提供 `/login`。
- 登录成功跳转 `/portal`。
- 未登录访问平台页面跳回 `/login`。

验收标准：

- 登录后第一屏是 `/portal`。
- 未登录不能访问业务页面和业务接口。

### EAI-M1-02：平台门户 `/portal`

实现范围：

- 展示平台名称、当前登录用户、当前组织名称、退出登录。
- 展示助手卡片：计划管理助手、AI 工作流中心、制度理解、采购、合同审查、人事、财务、会议纪要。
- 计划管理助手状态为“已上线 / 可用”。
- AI 工作流中心状态为“Demo 版”。
- 其他助手状态为“待建设 / 敬请期待”。
- 点击计划管理助手进入 `/assistants/plan`。
- 点击 AI 工作流中心进入 `/workflows`。
- 点击待建设助手不进入真实业务。

验收标准：

- `/portal` 不显示计划管理助手内部侧边栏。
- 领导能一眼识别这是企业管理 AI 助手平台。

### EAI-M1-03：平台总览 `/overview`

实现范围：

- 展示已上线助手数量、demo 助手数量、待建设助手数量。
- 展示当前组织、当前用户角色、计划管理助手运行状态、AI 工作流中心 demo 状态。
- 展示待建设助手列表、外部系统集成预留、平台架构图和一期建设范围。

验收标准：

- `/overview` 不显示计划管理助手内部侧边栏。
- 页面明确表达“平台不是单一计划管理系统”。

### EAI-M1-04：系统设置 `/settings`

实现范围：

- 顶部导航提供系统设置入口。
- 管理员可进入系统设置。
- 普通用户不能进入用户管理。
- 用户管理先做基础列表和创建入口。

验收标准：

- `/settings` 不显示计划管理助手内部侧边栏。
- 权限控制由后端兜底。

## 4. 身份权限任务

### EAI-M2-01：用户、角色、组织模型

实现范围：

- 角色至少支持 `leader`、`admin`、`member`。
- 用户关联组织。
- 当前用户接口返回角色、组织、权限摘要。
- 关闭公开注册，仅管理员创建用户。

验收标准：

- 管理员可创建用户。
- 普通用户不能创建用户。
- 当前用户接口返回角色和组织。

### EAI-M2-02：后端权限边界

实现范围：

- 普通用户只能访问和维护自己的计划/总结和格式检查报告。
- 领导可查看全员看板和报告。
- 管理员可用户管理、封存、解封、导出和全量维护。
- 前端权限控制只做体验，不作为安全边界。

验收标准：

- 未登录访问业务接口返回 401/403。
- 普通用户访问他人数据失败。
- 非管理员执行管理员操作失败。

## 5. 计划管理助手任务

### EAI-M3-01：计划管理助手入口

实现范围：

- `/assistants/plan` 是计划管理助手入口。
- 入口提供“返回助手门户”。
- 默认进入员工首页。

验收标准：

- 平台一级导航不直接暴露计划管理内部功能。

### EAI-M3-02：计划管理助手内部布局

实现范围：

- 仅 `/assistants/plan/*` 显示内部侧边栏。
- 菜单包括助手首页、我的计划/总结、领导看板、单人 PPT 上传、总 PPT 拆分、数据浏览、格式审查、总结转计划、计划转总结、PPT 导出、批量导出、数据封存。
- 管理员专属项按权限展示。
- 侧边栏支持隐藏/展开，并记住状态。

验收标准：

- `/portal`、`/overview`、`/workflows`、`/settings` 不显示该侧边栏。

### EAI-M3-03：员工默认首页

实现范围：

- 展示我的计划、我的总结、我的待处理问题、我的格式检查报告。
- 提供上传入口和数据维护入口。

验收标准：

- 普通用户进入 `/assistants/plan` 后只看到自己的数据摘要。

## 6. 计划管理核心业务任务

### EAI-M4-01：单人 PPT 上传解析

API 路径：`/api/assistants/plan/uploads/single/*`

实现范围：

- 上传单人计划 PPT 和单人总结 PPT。
- 填写责任人、日期批次、文件类型。
- 解析后先展示预览。
- 用户确认后入库。
- 同日期、同人员、同类型已有数据默认覆盖。
- 已封存日期禁止保存。

验收标准：

- 普通用户能上传自己的 PPT。
- 预览确认后入库。
- 封存后保存失败。

### EAI-M4-02：总 PPT 拆分入库

API 路径：`/api/assistants/plan/uploads/master/*`

实现范围：

- 管理员上传总 PPT。
- 填写日期批次。
- 按人员、板块拆分并展示预览。
- 管理员确认入库。
- 覆盖同日期、同人员、同类型旧数据。
- 已封存日期禁止覆盖入库。

验收标准：

- 普通用户不能执行总 PPT 入库。
- 管理员可完成拆分预览和确认入库。

### EAI-M4-03：计划/总结数据浏览维护

API 路径：`/api/assistants/plan/items/*`

实现范围：

- 按责任人、日期批次、类型筛选。
- 单条编辑、单条删除、按筛选条件清空。
- 已封存日期禁止新增、编辑、删除、清空、覆盖入库。

验收标准：

- 普通用户只能维护自己的数据。
- 管理员可维护全量数据。
- 封存日期所有普通写操作失败。

## 7. 报告、封存、审计、转换

### EAI-M5-01：格式检查报告入库

API 路径：`/api/assistants/plan/reviews/*`

实现范围：

- 覆盖计划和总结两类。
- 审查结果入库。
- 区分严重、警告、提示。
- 报告关联日期批次、责任人、类型、问题数量、问题明细、生成时间。

验收标准：

- 每次提交后可生成报告。
- 报告可列表查询和详情查看。

### EAI-M5-02：格式检查报告权限

实现范围：

- 员工只能看自己的报告。
- 领导和管理员可看全员报告。
- 点击跳转具体数据行可降级。

验收标准：

- 普通用户不能查看他人报告。

### EAI-M6-01：封存与解封

API 路径：`/api/assistants/plan/archives/*`

实现范围：

- 管理员可封存和解封日期批次。
- 封存后仍允许查看和导出。
- 非管理员不能封存/解封。

验收标准：

- 权限和状态变更正确。

### EAI-M6-02：全写入口封存校验

实现范围：

- 单人上传入库、总 PPT 覆盖入库、编辑、删除、清空、总结转计划保存、计划转总结保存都必须校验封存状态。

验收标准：

- 已封存日期下，上述所有写操作全部失败。

### EAI-M6-03：管理员强制修改与审计日志

API 路径：

- `/api/audit-logs/*`
- `/api/assistants/plan/items/force-update/*`

实现范围：

- 管理员强制修改已封存数据必须走专门入口。
- 审计日志记录操作者、时间、操作类型、影响对象、修改前后关键数据和原因。
- 提供审计日志查询。

验收标准：

- 强制修改后可查到审计日志。
- 普通写接口仍不能修改封存数据。

### EAI-M7-01：总结转计划

API 路径：`/api/assistants/plan/conversions/summary-to-plan/*`

验收标准：

- 转换结果可保存。
- 覆盖规则正确。
- 已封存日期禁止转换保存。

### EAI-M7-02：计划转总结

API 路径：`/api/assistants/plan/conversions/plan-to-summary/*`

验收标准：

- 转换结果可保存。
- 覆盖规则正确。
- 已封存日期禁止转换保存。

## 10. 每个任务的通用提交要求

每个任务完成后必须：

1. 执行 `git status`。
2. 确认不提交 `.env`、数据库、缓存、导出文件、`.venv`、`node_modules`、`.next`。
3. 后端任务执行：

```powershell
cd backend
.\.venv\Scripts\python.exe manage.py check
.\.venv\Scripts\python.exe manage.py makemigrations --check --dry-run
.\.venv\Scripts\python.exe manage.py test
```

4. 前端任务执行：

```powershell
cd frontend
npm.cmd install
npm.cmd run build
```

5. 提交并推送 GitHub。
6. 回复提交哈希和验证结果。

## 11. 额度上限处理

若研发 agent 遇到模型额度、API 额度、工具额度或平台额度上限：

1. 记录当前任务编号、已完成内容、未完成内容和报错。
2. 保存代码和文档改动。
3. 如达到可提交状态，先提交并推送 GitHub。
4. 暂停继续消耗额度。
5. 间隔 1 小时后重新启动一次会话或工具。
6. 重启后先执行轻量验证，确认额度是否重置。
7. 未重置则再等待 1 小时后重复验证。
